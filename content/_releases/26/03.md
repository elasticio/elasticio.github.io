## New Features
### Email notification update for New Component Versions
We have refined the email notification system for new component versions to ensure relevant users are informed. Now, only users who are actively using a component in a flow will receive email notifications when a new version of that component becomes available.
This update enhances the relevance of notifications, ensuring that only users impacted by the changes are alerted.

### New `/usage` endpoints in Quota Service
We have added new API endpoints to the quota service to provide detailed usage information:

* `GET /usages/:quota_type`: This endpoint returns all fields related to the specified quota type, excluding `counters`.
* `GET /usages/:quota_type/:context`: This endpoint provides usage information for a specific quota type and context.

These enhancements allow for more precise monitoring and management of quota usage, offering users greater insight into their resource consumption.

### API endpoint refactoring for Quota and Quota Usage
We have made significant improvements to our API endpoints related to quota and usage to enhance clarity and functionality:
* Endpoint Renaming:
    * The existing `/v2/quota-usage` endpoint has been renamed to `/v2/runtime-quota-usage` for better context and clarity.
* New API Endpoints:
    * `/v2/quotas/limits`: Provides information on quota limits.
    * `/v2/quotas/usages`: Redirects requests to the quota service usage, streamlining access to usage data.

For detailed information on all quota and usage endpoints, please refer to the updated API documentation:
* [Runtime Quota Usages](https://api.elastic.io/docs/v2#/runtime%20quota%20usages)
* [Quota Usages](https://api.elastic.io/docs/v2#/quota%20usages)
* [Quota Limits](https://api.elastic.io/docs/v2#/quota%20limits)

These changes are designed to improve the usability and organization of our API, ensuring you have efficient access to the information you need.

## Fixed Bugs
*   **FIXED** missing mapper passthrough on Retry from previous step
*   **FIXED** an issue regarding team membership transfers in contract management. Previously, if a contract owner removed a user who was the sole member of a developer team, the team membership did not transfer correctly. Now, when such a removal occurs, ownership of the team is automatically reassigned to the first contract owner.
